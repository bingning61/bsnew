# 第三章 运动控制与焊缝跟踪方法材料草稿

## 3.1 本章定位

本章必须围绕运动控制展开。与第二章强调“如何看见目标”不同，本章强调“如何根据目标位置偏差生成运动控制量，并使机器人形成焊缝跟踪闭环”。因此，本章的重点应当是偏差构建、控制量设计、速度协同调节、ROS1 闭环实现以及异常处理与安全停止逻辑。

在论文写作中，不能把本章写成“沿用了原来的控制代码”。更准确的写法应是：本文构建了一种基于图像中心偏差的焊缝跟踪控制方法，并在现有 ROS1 控制节点中实现了该方法。也就是说，代码复用只是工程实现形式，方法本身仍然应当以“偏差驱动控制策略”的方式来表述。

## 3.2 控制目标

根据当前系统逻辑，控制目标可以明确写成：在机器人前进过程中，使焊缝目标在图像中的中心位置尽可能接近图像中心，从而维持机器人相对于焊缝目标的对准状态。当目标偏差较小时，机器人保持较高前进速度；当目标偏差较大时，机器人增强转向修正并适当降低前进速度；当目标无法检测或检测超时时，机器人停止运动。

这一控制目标具有明显的视觉伺服特征。当前系统并未直接使用里程计或地图坐标作为焊缝跟踪误差，而是直接使用图像中的目标中心位置来构造误差，因此其主闭环是一种基于视觉位置偏差的运动控制闭环。

## 3.3 偏差构建方法

当前控制器在外部中心点模式下接收 `Point` 类型消息，其中 `Point.x` 为目标中心横坐标 `x_obj`，`Point.y` 为图像宽度 `W`，`Point.z` 为有效标志。控制器首先构造参考中心：

`x_ref = W / 2`

随后，把偏差等价写成：

`e = x_ref - x_obj`

如果用归一化形式表达，则当前代码可写为：

`e_n = (x_ref - x_obj) / x_ref`

应注意的是，`line_detector.py` 的 `twist_calculate(width, center)` 函数中，形参 `width` 实际上传入的是图像半宽，也就是参考中心 `x_ref`，而不是整幅图像宽度。因此，第三章在解释代码时应明确指出：当前控制器采用的是**相对于图像中心的归一化横向偏差**，而不是相对于整幅图像宽度的任意偏差定义。

这种偏差构建方式的优点在于简单而直接。由于控制器只关心目标中心相对于图像中心是“偏左”还是“偏右”，以及偏差量大还是小，因此采用单一横向偏差就能形成一套可运行的控制逻辑。

## 3.4 控制量设计

当前代码中的控制输出为 `geometry_msgs/Twist`，其主要使用 `linear.x` 与 `angular.z` 两个分量。具体来说，系统将前进速度写入 `linear.x`，将转向角速度写入 `angular.z`，而 `linear.y`、`linear.z`、`angular.x`、`angular.y` 全部置零。由此可见，当前焊缝跟踪方案采用的是“前进速度 + 偏航角速度”的控制形式。

这一设计非常适合论文表述。因为它说明系统并没有把视觉结果直接转成底盘私有协议，而是先生成标准的 ROS 运动指令 `cmd_vel`，再由底盘接口模块负责执行。换言之，控制器输出的是上层运动控制量，而不是底层驱动电机的原始参数，这使系统结构更加清晰。

需要特别说明的是，当前控制器不是 PID，也不是 Stanley、MPC 或滑模控制。根据代码，当前控制律本质上是一种简单的比例式偏差控制与分段速度调节策略。论文写作中应忠实于这一事实，不宜拔高为仓库中并不存在的高级算法。

## 3.5 运动控制逻辑

当前 `twist_calculate()` 的核心逻辑可以概括为两段。

第一段是目标对准判定。若目标中心与参考中心足够接近，即满足：

`0.95 < x_obj / x_ref < 1.05`

则控制器认为目标基本处于图像中心附近，此时不额外施加转向修正，直接令前进速度：

`v = 0.2`

第二段是偏差修正。当目标偏离中心时，控制器根据偏差生成角速度：

`ω = (x_ref - x_obj) / (2 x_ref)`

从这个式子可以看出，当前角速度与归一化偏差成正比，因此它可以被视为一种基于偏差的比例式转向调节。偏差越大，角速度绝对值越大；偏差越小，角速度绝对值越小。控制器并未引入积分项或微分项，因此结构简单、易调试，但控制精细度也相对有限。

## 3.6 速度协同调节逻辑

本项目第三章必须重点写速度协同调节，因为当前代码并非只计算角速度，还会根据转向需求调整前进速度。

在偏差修正阶段，控制器先计算 `ω`。若 `|ω| < 0.2`，说明当前偏差不算过大，控制器采用：

`v = 0.2 - ω / 2`

这一设计体现出明显的速度协同思想：当机器人需要轻微修正时，仍保持接近高速的前进状态；当转向量变化时，线速度也随之联动调整。若 `|ω| >= 0.2`，说明偏差较大，控制器直接把前进速度降为：

`v = 0.1`

从论文角度看，这可以抽象为一种**基于偏差大小的分段式线速度调节方法**。其核心思想是：偏差小时兼顾跟踪与效率，偏差大时优先保证修正能力和运行安全。

还应当指出一个真实代码细节：在 `v = 0.2 - ω / 2` 这一表达下，当前线速度不仅与角速度大小有关，也与角速度符号有关。因此，严格来说，这不是只依赖 `|ω|` 的纯对称速度调节。论文写作时可以如实写为“当前代码采用角速度耦合式线速度调节，其具体数值仍需通过实机测试进一步验证是否需要对称化优化”。

## 3.7 感知到控制的闭环机制

本项目的焊缝跟踪闭环可写成如下过程：相机采集当前图像后，YOLO 节点提取目标中心位置；控制节点读取该中心位置并生成 `cmd_vel`；底盘根据 `cmd_vel` 执行运动；机器人姿态变化又导致下一帧图像中的目标位置发生变化；新的图像再次进入感知节点，形成循环。

需要注意的是，当前闭环的主反馈量不是里程计，而是图像中的目标中心位置。`base_control.py` 虽然会发布 `odom`、`battery` 等话题，但这些反馈当前并没有直接进入焊缝跟踪控制律。因此，论文写作时应把当前系统定义为**以视觉位置偏差为主反馈量的闭环控制系统**。这种表述更符合当前代码事实，也有助于突出视觉伺服的特点。

## 3.8 异常处理与安全停止逻辑

当前系统在安全处理上具有两层保护。

第一层保护发生在控制节点。`line_detector.py` 在外部中心点模式下，若收到的 `Point.z` 表示无效检测，则立即调用 `publish_stop()` 发布零速度。除此之外，节点还启动了定时器看门狗，一旦在设定时间窗口内没有接收到新的有效中心点消息，也会发布零速度。默认超时时间在 `seam_tracking.launch` 中设为 `0.5 s`。

第二层保护发生在底盘接口层。`base_control/README.md` 中明确给出了下位机超时断连定义：如果建立连接后超过 `1000 ms` 未收到新的协议内数据，底盘将判定上位机断开连接并主动停止电机运动。虽然这一机制不是焊缝跟踪算法本身的一部分，但它与上层零速度输出共同构成了系统安全保障链条。

因此，本章完全可以把异常处理单独写成一个小节，标题可定为“目标丢失条件下的安全停止机制”。

## 3.9 控制输出到执行接口的实现路径

当前控制节点并不直接面对电机驱动器，而是通过 `cmd_vel` 与 `base_control.py` 解耦。`base_control.py` 接收到 `cmd_vel` 后，会将 `linear.x`、`linear.y` 和 `angular.z` 按比例转换为整数形式，并封装成串口协议帧，通过 `/dev/move_base` 发送到底盘控制板。与此同时，底盘接口还会周期性查询并发布里程计、电池信息以及可选的 IMU/超声波信息。

这一实现路径在论文中有两个价值。第一，它说明本项目的控制输出采用 ROS 标准接口，具有较好的工程可迁移性。第二，它说明原有底盘执行链路被较大程度保留，论文可以据此强调当前方案的低改动、低风险和易实现特点。

## 3.10 当前控制方法的局限性

第三章在结尾处还应诚实指出当前控制方案的局限性。

首先，当前控制误差只使用了单一横向中心偏差，没有引入焊缝方向、曲率或连续轨迹信息，因此对复杂焊缝形态的描述能力有限。其次，当前控制律采用简单比例式调节，未引入滤波、速度平滑或预测机制，对噪声和快速变化场景的鲁棒性仍需实机验证。再次，当前方案只使用 `linear.x` 和 `angular.z`，即使机器人底盘具备全向运动能力，也没有利用横向速度 `linear.y` 进行更直接的横向修正。最后，线速度与角速度之间的耦合关系是经验式的，是否适合实际工况仍需通过实验进一步标定。

## 3.11 本章结尾可直接使用的总结段

本章围绕焊缝跟踪过程中的运动控制问题，构建了基于图像中心偏差的速度生成方法。系统在接收视觉节点输出的目标中心位置后，以图像中心作为参考构造归一化偏差，并依据偏差大小生成角速度，同时通过分段式线速度调节实现转向与前进的协同控制。在系统实现上，控制器输出标准 `cmd_vel` 指令，由原有底盘接口模块完成底层执行；在安全机制上，系统通过无效检测零速度输出、中心消息超时看门狗以及下位机超时停机机制共同保障运行安全。由此，当前项目形成了一条从视觉偏差到运动控制量的完整技术链路，为焊缝跟踪闭环的实现提供了方法基础。
