# 本科毕业论文单文件总交接包

## 1. 项目定义

当前项目最适合被定义为：**基于 YOLO 视觉感知、检测框中心位置表征与中心偏差控制的 ROS1 焊缝跟踪机器人系统**。

该系统的基本目标是：从相机图像中识别焊缝目标，提取目标在图像中的中心位置，构造目标中心与图像中心之间的偏差，再根据偏差生成 `cmd_vel`，驱动机器人沿焊缝方向进行稳定跟踪，并在无检测或检测中断时安全停止。

## 2. 核心问题

这个项目真正要解决的问题不是“如何把两个现成系统接起来”，而是：

1. 如何利用视觉方法稳定识别焊缝目标。
2. 如何把检测结果转换为控制器可用的几何位置量。
3. 如何构建目标中心与图像中心之间的偏差关系。
4. 如何依据偏差生成运动控制量并形成 ROS1 闭环。

因此，整篇论文的主线应当是“感知目标、表征位置、构造偏差、生成控制、形成闭环、验证可行性”。

## 4. 当前主运行路径

当前主入口是：

`catkin_ws/src/robot_vision/launch/seam_tracking.launch`

它组织出的主数据流为：

```text
/image_raw
-> yolo_seam_detector.py
-> /seam_center (geometry_msgs/Point)
-> line_detector.py
-> cmd_vel (geometry_msgs/Twist)
-> base_control.py
-> /dev/move_base
-> chassis motion
-> next image frame
```

这条链路说明，系统已经形成了一条从视觉感知到底盘执行的闭环路径。

## 5. YOLO 在系统中的功能

YOLO 在当前系统中承担的是**视觉感知前端**功能，而不是整个系统的全部功能。它负责从 `/image_raw` 图像中检测焊缝目标，并从检测结果中选出一个最优检测框。当前代码默认采用最高置信度策略筛选最优框；如果设定了 `target_class_id`，则会先进行类别过滤。

随后，YOLO 节点并不把整套检测框直接传给控制器，而是提取检测框中心横坐标 `x_obj=(x1+x2)/2`，同时记录图像宽度 `W`，再把两者封装为 `geometry_msgs/Point`：

- `Point.x = x_obj`
- `Point.y = W`
- `Point.z = valid`

因此，YOLO 的核心作用可以概括为：**完成目标检测，并把语义检测结果转换为控制可用的几何位置输入**。

## 6. 控制器在系统中的功能

控制器由 `robot_vision/scripts/line_detector.py` 承担。虽然该脚本历史上还包含 HSV 阈值分割路径，但在当前主链路中，它以外部中心点模式工作，不再依赖图像分割，而是直接读取 `/seam_center`。

控制器的核心功能包括：

1. 根据图像宽度计算参考中心 `x_ref=W/2`
2. 根据目标中心 `x_obj` 构造偏差
3. 根据偏差生成角速度 `angular.z`
4. 根据转向需求协同调节线速度 `linear.x`
5. 在无效检测或消息超时时发布零速度

因此，控制器承担的是**偏差构建与速度生成**功能，而不是视觉识别功能。

## 7. 视觉信息如何转换成运动控制量

这一转换过程是整篇论文最重要的逻辑链条，可以概括为：

1. YOLO 得到最优检测框。
2. 系统计算检测框中心横坐标 `x_obj`。
3. 系统利用图像宽度 `W` 构造参考中心 `x_ref=W/2`。
4. 系统建立偏差 `e=x_ref-x_obj`。
5. 系统依据偏差生成角速度：

`ω = (x_ref - x_obj) / (2x_ref)`

6. 系统再依据转向量调节线速度：

- 若目标基本居中，则 `v=0.2`
- 若偏差存在且 `|ω|<0.2`，则 `v=0.2-ω/2`
- 若偏差较大，则 `v=0.1`

7. 最终输出 `cmd_vel`。

这说明当前系统的本质不是“检测后直接输出控制”，而是“检测 -> 位置表征 -> 偏差中间量 -> 速度生成”。

## 8. 当前系统的核心方法体系

从论文方法角度，当前项目可以被组织成以下方法体系。

### 8.1 视觉目标感知方法

采用 YOLO 对图像中的焊缝目标进行检测，并在候选目标中选出最适合当前控制的最优检测框。

### 8.2 目标位置表征方法

采用检测框中心横坐标作为目标位置表征量，并把图像宽度作为参考中心构造所需的辅助量。

### 8.3 偏差构建方法

以图像中心为期望位置，把目标中心与图像中心之间的差值写成归一化偏差，为控制器提供统一输入。

### 8.4 运动控制方法

依据归一化偏差生成转向角速度，并以标准 `cmd_vel` 形式输出控制量。

### 8.5 速度协同调节方法

根据偏差大小对线速度进行分段调节，使系统在对准较好时保持较快前进，在偏差较大时优先修正姿态。

### 8.6 感知到控制闭环实现方法

通过 ROS1 话题把视觉节点、控制节点与底盘接口解耦，形成从 `/image_raw` 到 `cmd_vel` 的闭环链路。

### 8.7 异常处理与安全停止方法

通过检测有效标志和中心消息超时机制，在目标丢失时输出零速度；底盘接口文档还给出了下位机超时停机机制。

### 8.8 模型与仿真支撑方法

通过 `nanoomni_description`、`robot_simulation` 和 `robot_navigation` 提供机器人模型、Gazebo/Stage 环境和通用移动平台支撑。

## 9. 当前系统完整逻辑链条

整套系统最适合写成如下逻辑链条：

```text
相机采图
-> YOLO 检测焊缝目标
-> 选择最优检测框
-> 提取检测框中心横坐标
-> 以图像中心构造偏差
-> 根据偏差生成角速度
-> 根据偏差和角速度协同调整线速度
-> 输出 cmd_vel
-> 底盘执行
-> 新图像反馈
```

这条逻辑链条非常适合写成论文第二章和第三章的连接桥梁。

## 10. 为什么这种路线适合本科毕业设计

这种路线适合本科毕业设计，原因在于它兼顾了方法清晰和工程可实现性。

首先，系统问题明确，技术路线不分散，容易形成一篇主线清楚的论文。其次，视觉与控制通过最小几何接口耦合，避免了大规模重构已有底盘链路。再次，方法足够“像论文”，因为它不仅有工程实现，还能抽象出目标感知、位置表征、偏差构建、速度调节和闭环实现等明确的方法模块。最后，系统仍然保留了足够多的后续改进空间，便于在第五章写出合理的展望。

## 12. 当前系统的局限性

当前方案存在如下局限：

1. 仓库中没有实际 `.pt` 权重文件。
2. 训练数据集与标签映射未随仓库提供。
3. 当前位置表征只使用检测框中心，信息粒度较粗。
4. 当前控制律为简单比例式偏差控制，不是高级控制算法。
5. 当前线速度与角速度耦合策略仍需实机标定。
6. 当前主链路混用了 `python3` 与 `python` 风格脚本，环境配置需人工确认。
7. 当前缺少完整实机实验数据、截图与定量指标。
8. 当前没有焊缝专用仿真场景与专用联调 launch。

## 15. 回到原环境后的构建与启动方式

通用构建方式：

```bash
cd ~/bsnew/catkin_ws
source /opt/ros/<ros1_distro>/setup.bash
catkin_make
source devel/setup.bash
```

常见 Ubuntu 18.04 ROS1 场景可写为：

```bash
cd ~/bsnew/catkin_ws
source /opt/ros/melodic/setup.bash
catkin_make
source devel/setup.bash
```

主运行命令：

```bash
roslaunch robot_vision seam_tracking.launch \
  run_base_control:=true \
  model_path:=../yolo/runs/train/exp17/weights/best.pt
```

仅调试感知与控制链路：

```bash
roslaunch robot_vision seam_tracking.launch \
  run_base_control:=false \
  use_fake_camera:=true \
  fake_image_path:=$(rospack find robot_vision)/data/bingda.png \
  model_path:=../yolo/runs/train/exp17/weights/best.pt
```

需要明确说明：当前仓库中没有实际 `best.pt` 文件，上述路径只是 README 与脚本注释中的示例，必须由人工替换为真实权重。
